## 1. ISOLAMENTO DATI — la tua paura n.1 → **design CORRETTO** (verificato sul DB live)
Ho letto le **policy RLS reali** del database di produzione. Sono scritte bene: ogni tabella utente è isolata per `auth.uid()`. Nessun `USING (true)` su dati personali.

| Risorsa | Policy sul DB live | Esito |
|---|---|---|
| **Carrello** `user_carts` | `auth.uid() = user_id` (read+write) | ✅ B non vede/scrive il carrello di A |
| **Ordini** `orders` | SELECT: proprio (`user_id`) · proprio seller (`seller_id`) · rider assegnato/disponibile · admin | ✅ B non vede ordini di A; S2 non vede ordini di S1 |
| **Righe ordine** `order_items` | via ownership ordine / prodotti del seller / rider assegnato / admin | ✅ |
| **Indirizzi** `user_addresses` | `user_id = auth.uid()` | ✅ |
| **Chat** `conversations`/`messages` | solo partecipanti (`buyer_id`/`seller_id`) | ✅ |
| **Resi** `returns` | buyer / seller / admin | ✅ |
| **Gift card** `gift_cards` | `buyer_id` o `redeemed_by` | ✅ |
| **Loyalty / Push** | `user_id = auth.uid()` | ✅ |
| **Codici** `order_pickup_codes` / `order_delivery_codes` | pickup → **solo seller** dell'ordine/admin · delivery → **solo buyer** dell'ordine/admin | ✅ (il rider verifica via RPC, non legge i codici) |
| **Profili** `profiles` | own / seller approvato / opt-in pubblico / rider-di-ordine / admin | ✅ nessuna lettura pubblica della PII |

**Conferma del fix 🔴-1:** le policy `profiles` confermano che un profilo buyer/rider **non** è leggibile da `anon` → il fix che legge il profilo via service-role era **necessario e corretto**. ⚠️ ma deve essere **DEPLOYATO** in produzione (vedi condizioni).

**Cosa NON ho potuto fare:** il test comportamentale "login come B → apri l'ordine di A → 403/vuoto" con dati reali (serve test env o tuo OK). Il design però è solido.

## 2. SICUREZZA CARTA (PCI) → 🟢
- **La carta non tocca mai il nostro backend.** Il pagamento usa **Stripe Checkout hosted** (`lib/stripe/client.ts`: `checkout.sessions.create({ mode:'payment', ... })` che ritorna l'`url` Stripe). PAN e CVV si inseriscono **sul dominio Stripe**.
- **Nessun dato carta persistito**: nel codice/privacy solo "ultime 4 cifre", **mai** numero completo né CVV.
- **Webhook firmato**: `getStripe().webhooks.constructEvent(raw, sig, STRIPE_WEBHOOK_SECRET)`.
- **Profilo PCI realistico: SAQ-A** (campo carta gestito interamente da Stripe).
- ⚠️ da confermare nel **Dashboard Stripe**: endpoint webhook configurato col secret giusto + HTTPS in prod.

## 3. SECURITY ADVISOR Supabase (DB live) → nessun bloccante
- **Nessun ERROR/CRITICAL.**
- INFO `rls_enabled_no_policy` su tabelle interne (`cron_heartbeats`, `email_queue`, `stripe_event_log`, `kpi_snapshots`, `merchants_leads`, `outreach_events`, `telegram_chats`, `uptime_checks`): RLS attiva **senza policy = deny-all** → accessibili solo via service-role → **sicuro**.
- WARN funzioni `SECURITY DEFINER` callable da `authenticated` (cancel_order, redeem_gift_card, confirm_cod_remittance, verify_pickup/delivery_code, admin_list_user_emails, …): **by-design**, con **guardia interna** verificata — es. `admin_list_user_emails` fa `IF NOT is_admin() THEN RAISE 'forbidden'` (un non-admin NON ottiene le email). Mitigati.
- WARN minori: `pg_trgm` in schema `public`; **leaked-password protection disabilitata** (consiglio: attivala, 1 click).
